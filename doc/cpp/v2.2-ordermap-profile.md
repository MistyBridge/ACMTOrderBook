## 2. orderMap 操作频率分析

### 2.1 调用点统计

| 操作 | 调用位置 | 触发频率 | 说明 |
|------|----------|----------|------|
| `insertOrderMap` | `axob_order.cpp:130` | ~50% msg | 每条 Order 消息 1 次 |
| `orderMap[seq] = p` | `insertOrderMap` 内部 | ~50% msg | **隐式 find + 可能 rehash** |
| `orderMap.find(seq)` | `onCancel:123` | ~10% msg | 撤单路径 |
| `orderMap.find(seq)` | `tradeLimit:131` | ~80% msg × 2 | 每笔成交 2 次（买卖双方） |

### 2.2 热点函数排序（按调用频率）

```
1. orderMap.find()  — 最热，每条 Trade 消息调 2 次，每条 Cancel 调 1 次
2. insertOrderMap   — 每条 Order 消息调 1 次
3. eraseOrderMap    — 当前未使用（never called）
```

### 2.3 消息比例估算（典型交易日，单只股票）

```
Order 消息:  ~40-50%  → insertOrderMap + orderMap[] = p（含隐式 find）
Trade 消息:  ~40-50%  → 2× orderMap.find() + 2× levelDequeue
Cancel 消息: ~5-10%   → orderMap.find() + levelDequeue
```

**结论：`orderMap.find()` 是绝对热路径**，每条消息平均调用 1.5-2 次。

## 3. 当前实现的性能问题

### 3.1 问题1：insertOrderMap 双重哈希查找 🔴

```cpp
void insertOrderMap(uint64_t seq, const ObOrder& order) {
    ObOrder* p = orderPool_->alloc();   // ① 内存分配
    *p = order;                          // ② 值拷贝（32B）
    orderMap[seq] = p;                   // ③ ⚠️ 双重哈希！
}
```

**`orderMap[seq] = p` 分解：**
1. `operator[](seq)` → 内部调用 `find(seq)` → 哈希计算 + 桶遍历
2. 未找到 → `emplace(seq, ObOrder*)` → 再次哈希计算 + 插入

**开销：** 2次哈希计算 + 2次桶遍历，每次约 20-40ns（含 cache miss 时更高）

**修复方案：** 用 `emplace` 替代 `operator[]`

```cpp
// 优化后：单次哈希
orderMap.emplace(seq, p);
```

### 3.2 问题2：std::unordered_map 的内存布局 🟡

```
std::unordered_map 内存结构：
┌─────────────┐
│ bucket array │ → 指针数组，每个 bucket 指向链表头
│ [0] → n1 → n2 → nullptr
│ [1] → nullptr
│ [2] → n3 → nullptr
│ ...
│ [N-1] → n4 → nullptr
└─────────────┘

每个节点 n：
┌───────────────┐
│ hash (size_t) │  8B  ← MSVC STL 额外存储 hash 值用于快速比较
│ next (Node*)   │  8B  ← 指向下一个节点（链表法）
│ key (uint64_t) │  8B
│ value (ptr)    │  8B
├───────────────┤
│ total: 32B     │  ← MSVC STL，GCC/Clang 略不同
└───────────────┘
```

**问题：**
- **指针追逐**：每个节点是独立堆分配，地址不连续，CPU 预取器无法工作
- **cache miss**：桶遍历每跳 1 次指针 = 1 次潜在 L1 miss（~4ns）
- **内存碎片**：82K 个节点 = 82K 次独立 malloc，内存碎片化
- **空间放大**：每个条目实际占用 ~64-80B（含 hash + next + key + value + 分配器开销）

### 3.3 问题3：未预分配容量 🟡

```cpp
// 当前：随用随长，触发多次 rehash
std::unordered_map<uint64_t, ObOrder*> orderMap;
// 默认 bucket count = 初始很小（如 10）
```

**典型股票一日 orderMap 规模：** ~20K-80K 条目  
**rehash 触发：** 每当 `size() > bucket_count() * max_load_factor()` 时全量 rehash  
**开销：** rehash 需遍历所有节点重新分配 bucket，O(n) 且阻塞

## 4. 内存分配模式分析

### 4.1 insertOrderMap 分配路径

```
insertOrderMap(seq, order)
  ├── orderPool_->alloc()     ← 内存池分配，O(1) 无锁
  │   └── freeList_ 摘头      ← 若空则 allocateBlock（2倍扩容）
  ├── *p = order              ← 32B memcpy
  └── orderMap[seq] = p       ← unordered_map 内部:
      ├── bucket[hash(seq)]   ← 计算桶位置
      ├── 遍历链表查找 seq    ← 通常 0-1 跳（新 key 不存在）
      └── new Node(hash,key,value,next) ← ⚠️ 每次插入都 malloc！
```

**关键瓶颈**：即使 ObOrder 用 MemoryPool 无堆分配，`unordered_map` 自身的节点仍需 `new Node(...)` → **每条 Order 消息触发 1 次堆分配**

### 4.2 eraseOrderMap：当前未使用

```cpp
// axob_trade.cpp:235
// 注意：Python 版本从不从 orderMap 删除订单！
// orderMap 保留所有历史订单，用于后续成交查找。
```

**这是有意设计**：orderMap 只增不删。  
**影响**：
- ✅ 无删除开销
- ✅ 容量稳定后无 rehash
- ⚠️ 内存持续增长（但有上限：单日消息总量）

## 5. Hash Collision 分析

### 5.1 Key 分布特征

```
ApplSeqNum (uint64_t)：
  - 深交所：单调递增，从 1 开始
  - 上交所：类似递增
  - 全市场共享序列号空间
```

### 5.2 std::unordered_map 默认哈希

```cpp
// MSVC: 对 uint64_t 使用 identity hash（直接返回值本身）
// GCC:  std::hash<uint64_t> → 折叠 + 乘法混合
```

**MSVC 问题**：identity hash + power-of-2 bucket count → 只用低位选择桶  
**示例**：bucket_count = 1024 时，hash(x) & 1023 只看低 10 位  
**冲突率**：如果 ApplSeqNum 低 10 位有规律（如步长=2），冲突率可达 50%

### 5.3 平铺哈希表的优势

| 方案 | 哈希策略 | 冲突处理 | 预计冲突率 |
|------|----------|----------|------------|
| `std::unordered_map` | 默认 hash（可能 identity） | 链表法 | 5-15%（MSVC） |
| `phmap::flat_hash_map` | SwissTable（SSE2 探测） | 开放寻址 + SIMD | <1% |
| `ankerl::unordered_dense` | wyhash + 开放寻址 | 线性探测 | <2% |
| `robin_hood` | 高质量 hash + Robin Hood | 开放寻址 | <1% |

### 7.2 柯1 平铺哈希表替换（本迭代目标）🎯

**选型关注点（针对 orderMap 特殊场景）：**

| 维度 | 关注点 |
|------|--------|
| **只增不删** | 某些哈希表对 tombstone 有额外开销，orderMap 不受影响 |
| **单调递增 key** | 需要好哈希函数避免低位聚集（SwissTable 天然处理好） |
| **内存开销** | 82K × 32B 指针值 = 2.6MB 值数据，额外开销需控制在 2× 以内 |
| **find 密集** | 查找性能是第一优先级（insert 第二） |

### 7.3 后续优化（v2.4+）🔭

| 优化 | 前置条件 | 预计收益 |
|------|----------|----------|
| SoA 替代 AoS（将 ObOrder 拆分为并行数组） | 大改动 | SIMD 友好，+5-10% |

## 8. 硬件特性洞察

### 8.1 CPU Cache 影响估算

```
orderMap 规模: ~82K 条目
每个节点: ~32-64B（含 hash + next + key + value + 分配器头）

std::unordered_map（链表法）:
  - 顺序插入时节点大致连续（malloc 追加分配）
  - 但随机查找时节点地址分散
  - L1 cache line = 64B → 每次 miss 浪费 1 cache line
  - 82K 节点 × 随机访问 → ~4ns/cache-line × 平均 0-1 跳 = ~4-8ns（理想）
  - 但 MSVC STL 节点 48B → 1 cache line 不够 → 可能 2 次 miss

flat_hash_map（SwissTable）:
  - 控制字节数组连续（每个条目 1B 状态 + 7B hash）
  - SSE2 SIMD: 一次比较 16 个控制字节 → 快速探测
  - 数据数组连续 → 预取友好
  - 82K 条目 × 40B = 3.2MB → 大部分在 L3 cache 中
```

### 8.2 分支预测影响

```cpp
// orderMap.find() 热路径分支：
auto it = orderMap.find(seq);
if (it != orderMap.end()) {  // 分支预测器会学到：几乎总是 true
    // ...                         预测准确率 >99%
}
```

平铺哈希表的开放寻址可能引入更多分支（探测循环），但现代 CPU 有足够 BTB 条目处理。

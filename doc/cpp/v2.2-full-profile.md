## 2. Component-by-Component Analysis

### 2.1 Producer: parseKeyValueLine() - BOTTLENECK [RED]

```cpp
// msg_util.cpp:6
std::unordered_map<std::string, int64_t> parseKeyValueLine(const std::string& line) {
    std::unordered_map<std::string, int64_t> dict;  // HEAP ALLOCATION per line!
    std::string key = line.substr(keyStart, pos - keyStart);  // HEAP per key
    std::string valStr = line.substr(valStart, pos - valStart);  // HEAP per value
    dict[key] = std::stoll(valStr);  // HASH + INSERT
    return dict;  // MOVE
}
```

**Problems:**
1. Creates NEW `std::unordered_map` for EVERY message line (~200K times)
2. `substr()` creates new `std::string` for each key/value (~10 per line)
3. ~10 hash computations + 10 allocations per message
4. Destructor deallocates all nodes after `loadDict()` reads values

**Cost estimate:** ~200-500ns per message = 4-10% of total time

**Optimization A: Reuse Map (Quick Win)**
```cpp
static thread_local std::unordered_map<std::string, int64_t> dict;
dict.clear();
parseKeyValueLineInto(line, dict);
```
Savings: ~100-300ns/msg

**Optimization B: Direct Parse (Bigger Win)**
```cpp
// Parse directly into message struct, skip intermediate map entirely
void AxsbeFileReader::advance() {
    std::string line;
    while (std::getline(file_, line)) {
        int msgType = extractIntValue(line, "MsgType");
        if (msgType == MsgType_order) {
            currentOrder_.ApplSeqNum = extractIntValue(line, "ApplSeqNum");
            currentOrder_.Price = extractIntValue(line, "Price");
        }
    }
}
```
Savings: ~200-450ns/msg (eliminates ALL intermediate allocations)

### 2.2 Consumer: Latency Measurement Overhead [YELLOW]

```cpp
// consumer.cpp:78-80
uint64_t dequeueTime = now_ns();          // ~20-50ns (Windows high_resolution_clock)
uint64_t latencyNs = dequeueTime - batch[i].enqueueTimestamp;
latency.record(latencyNs);               // ~5-10ns
```

**Total**: 2x `now_ns()` + `record()` = ~45-110ns per message = 1-2.3% of total

**Optimization: Sample-Based Latency**
```cpp
if ((totalConsumed & 0xF) == 0) {  // 1-in-16 sampling
    uint64_t dequeueTime = now_ns();
    latency.record(dequeueTime - batch[i].enqueueTimestamp);
}
```
Savings: ~40-100ns/msg (amortized)

### 2.3 Consumer: genSnap() Heap Allocation [YELLOW]

```cpp
void AXOB::genSnap() {
    AxsbeSnapStock snap;
    // ... fill snap ...
    delete lastSnap;                    // HEAP DEALLOC every message
    lastSnap = new AxsbeSnapStock(snap);  // HEAP ALLOC every message
}
```

**Problem**: `new` + `delete` per message = ~50-100ns heap overhead.

**Optimization**: Use stack member instead of heap pointer.
```cpp
// Change: AxsbeSnapStock* lastSnap = nullptr;
// To:     AxsbeSnapStock lastSnap_;
```
Savings: ~40-80ns/msg

### 2.4 SPSC Queue: Non-Trivially-Copyable MarketEvent

MarketEvent has non-trivial union members, so SPSC uses placement new instead of memcpy.

**Fix**: Make MarketEvent trivially copyable with raw char[] payload.
Savings: ~5-15ns per queue operation

### Key Insight: File I/O Dominates

```
Total wall time per message: ~4700ns (at 212K msg/s)

File I/O (producer):          ~2000ns  (43%)  <-- BIGGEST
CPU processing (consumer):    ~1000ns  (21%)
Queue wait / sync:            ~500ns   (11%)
Overhead (latency, genSnap):  ~200ns   (4%)
Other:                        ~1000ns  (21%)
```

**The producer is I/O-bound.** It spends ~2us per message reading the .log file,
while the consumer spends ~1us on processing. Optimizing consumer code has
diminishing returns until we also speed up the producer.

## 6. Conclusion

**The biggest bottleneck is NOT in AXOB core logic** (orderMap, LevelBook).

**The real bottlenecks are:**
1. **File I/O in producer** (~43% of wall time) - architectural
2. **parseKeyValueLine() heap allocations** (~6.4%) - easy fix
3. **genSnap heap allocation** (~1.5%) - easy fix
4. **Latency measurement overhead** (~1.5%) - moderate fix

**v2.2 target (240K msg/s = +13%) is achievable with Phase 1 optimizations.**

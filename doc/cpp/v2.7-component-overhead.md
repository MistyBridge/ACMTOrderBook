## 4. Top Bottlenecks

### #1: extractField strstr scanning (72.9%)

Each extractField call scans the entire ~500B line from the beginning:
- ORDER: 8 calls x ~57ns = ~456ns
- EXE: 9 calls x ~57ns = ~513ns
- SNAP: 57 calls x ~57ns = ~3249ns (rare)
- Weighted average: ~621ns/msg

**Fix**: Single-pass parser (scan once, extract all fields)
**Expected gain**: 621ns -> ~200ns = +49%

### #2: next() triple copy (6.8%)

Copies ALL 3 structs regardless of actual message type:
- order (~100B) + exe (~80B) + snap (~400B) = ~580B
- 2 out of 3 are wasted

**Fix**: Conditional copy (only actual type)
**Expected gain**: ~39ns/msg saved

### #3: MarketEvent 500B copy (12.6%)

- try_push: 500B memcpy = ~58ns
- pop_batch: 500B memcpy = ~50ns
- Total: ~108ns/msg

**Fix**: Reduce MarketEvent size
**Expected gain**: ~50-80ns/msg saved

### #4: Latency sampling (1.9%)

- now_ns() (QPC) ~10ns + record() ~5ns = ~16ns/msg
- Called on EVERY message

**Fix**: Sample every 8th message (DONE in v2.8)
**Expected gain**: ~14ns/msg saved
